# Codebase Review — 2026-06-10

Full-solution review at commit `ba24cf6` (clean working tree on `main`). Four parallel reviewers covered the API/Domain layers, the ServiceDefaults payload pipeline, the messaging/infrastructure projects, and security. Each read `docs/ARCHITECTURE_REVIEW.md` first, so nothing below re-reports previously accepted risks. The headline: the publish side and the classic security surface are genuinely hardened — the residual problems are concentrated **downstream of the Service Bus topic and in places the in-memory test doubles can't see**.

## High severity

**1. Events can be silently destroyed when a subscriber is down >1 hour** — `src/StarterApp.AppHost/ServiceBusTopology.cs:7`, `Program.cs:35,43,61` *(verified directly)*
Topic and both subscriptions set `DefaultMessageTimeToLive = 1h`, and `DeadLetteringOnMessageExpiration` is never enabled (Azure default is off — zero references in the repo). If the Functions worker is down or crash-looping for over an hour, every event in that window is deleted with no trace, while the outbox row is already marked processed. This quietly defeats the "no event permanently lost" posture the publish side worked hard for (#79, FailClosed). Integration tests never exercise consumer downtime because the emulator and Functions container always run together.

## Medium severity

**2. No retry policy in the Functions host → a seconds-long blob outage dead-letters live events** — `src/StarterApp.Functions/host.json:11-18`
Functions run with `PayloadCapture__ServiceBusFailureMode=FailClosed`, so a capture failure throws and the message is abandoned. Service Bus redelivers abandoned messages immediately; with `MaxDeliveryCount = 5` and no `retry` block in host.json, all five deliveries burn in seconds and the message lands in a DLQ nothing monitors. The same outage on the publish side deliberately pauses the batch with retry budget intact — the consume side is asymmetric.

**3. Outbox batch-final save loses all outcomes if one claim is stolen** — `src/StarterApp.Api/Infrastructure/Outbox/OutboxProcessor.cs:162` *(found independently by two reviewers)*
All `MarkAsProcessed`/`IncrementRetry`/`MarkAsError` outcomes for a 20-message batch persist in one `SaveChangesAsync`, and `ProcessingId` is a concurrency token. If one row's 60s lock expires mid-batch (plausible: ~10+ serial blob round trips per message plus the SB send) and another replica reclaims it, the save throws and **every** outcome in the batch is discarded — everything already published gets republished after lock expiry, outside the 5-minute dedup window. Per-row saves or catching `DbUpdateConcurrencyException` and persisting the rest would close it.

**4. Missing `ConnectionStrings:servicebus` silently disables eventing in production** — `src/StarterApp.Api/Infrastructure/ServiceCollectionExtensions.cs:172-174`
The no-op publisher fallback is intentional for tests, but there's no production guard like the one `GatewayIdentity:Mode` has: a typo'd env var boots green, passes `/health/ready` (DB-only), and accumulates outbox rows forever. An environment-gated `ValidateOnStart` or an outbox-depth health check would fail this loudly.

**5. Payload capture runs before rate limiting and gateway auth** — `src/StarterApp.Api/Program.cs:68`
`UsePayloadCapture()` is the first middleware, so every request — including 401/403/429 rejections and 404 junk — is fully archived: archive + audit + up to 64 entity-index blob appends, all serial on the request thread. Archive volume/cost is bounded only by the upstream gateway, and a rate-limited caller still amplifies each request into ~60+ storage calls. If capturing rejected traffic isn't a hard compliance requirement, move capture behind the limiter or add a capture-path throttle — either way it deserves a recorded decision.

**6. Retention cleanup timer may never fire in the orchestrated path** — `src/StarterApp.Functions/PayloadArchiveCleanupFunction.cs:28` *(medium confidence — needs a real-host check)*
The trigger is `%PayloadCapture__CleanupCron%`, but the env-var configuration provider normalizes `__` to `:`, so the name resolver's lookup of the literal double-underscore key plausibly returns null. Supporting evidence: `PayloadCaptureOptions.CleanupCron` is dead code (nothing binds it), and the only cleanup test calls `RunAsync` directly, bypassing trigger resolution. Worth verifying once against a real Functions host; if it does resolve, a test pinning that would be cheap. Related: if the setting is genuinely absent in a deployment, the failed timer indexing can take down the Service Bus subscribers in the same worker.

**7. Sensitive `*Id` values bypass redaction into blob paths and Information logs** — `src/StarterApp.ServiceDefaults/Payloads/PayloadEntityReferenceExtractor.cs:113-135`
Any scalar property ending in `Id` becomes an entity reference without consulting `SensitivePropertyNames`: `{"nationalId": "..."}` produces `entity-index/national/<value>/...`, which the sink then logs verbatim in `EntityIndexBlobNames` — violating the "logs remain redacted" rule. The pinned pointer-only test uses `email`, which doesn't end in `Id`, so nothing catches it. (It also generates junk references from words like `paid`.)

**8. Concurrent appenders can corrupt >4 MiB JSONL lines in the shared audit blob** — `src/StarterApp.ServiceDefaults/Payloads/AzureBlobPayloadArchiveStore.cs:26-39`
The U9 chunking fix preserves one writer's block order, but with no append-position condition another request's block can land between chunks of an oversize line in the shared per-minute audit blob, splicing two records. Only bites when `MaxPayloadBytes` > ~4 MiB (the `[Range]` allows 100 MiB), and no test can see it — the in-memory store appends atomically under a lock.

**9. All 16 endpoint handlers sever the cancellation chain** — `src/StarterApp.Api/Endpoints/*.cs`
No route handler injects `CancellationToken`, so the carefully threaded token plumbing downstream (Dapper `CommandDefinition` tokens from finding #32, EF, `PostgresRetryPolicy`) always gets `CancellationToken.None`. A disconnected client can't abort anything — under DB degradation, the stacked retry policies keep burning connections for minutes per dead request, and the `OperationCanceledException → 499` mapping is unreachable. The repo's own exemplar skill (`.claude/skills/api-design/SKILL.md:43-53`) shows the token being forwarded; this is doc/code drift a convention test could enforce mechanically.

**10. Retry-idempotency fix applied to orders but not the other aggregates** — `CreateProductCommand.cs:36`, `CreateCustomerCommand.cs:34`
CLAUDE.md documents the commit-ambiguity failure mode and fixes it only for `CreateOrderCommandHandler`. A commit-succeeded-but-ack-lost retry gives Product a silent duplicate row (no unique constraint) and Customer a spurious 409 for a create that succeeded; re-run updates/deletes produce false `xmin` 409s. Neither tests nor the review doc record this as accepted for the other aggregates.

## Lower severity

- **Price has no upper bound** → `1e17` passes validation, hits `numeric(18,2)` overflow (SqlState 22003), unmapped → client-input-driven 500. Same class as fixed finding U4. (`CreateProductCommandValidator.cs:15`)
- **`{"status": 99}` binds as an undefined enum** → misrouted to the domain guard → 409 with a leaky `"Cannot transition from Pending to 99"` detail instead of 400. The validator half of the Validator–Domain Guard Sync Rule is missing here. (`UpdateOrderStatusCommandValidator.cs:10`)
- **In-memory archive store cleanup diverges from the Azure store three ways** (hardcoded prefixes, deletes outside prefixes, no batch cap) — all cleanup tests run against the double, so none of it is detectable. (`InMemoryPayloadArchiveStore.cs:43-67`)
- **`RequestFailedException` with `Status == 0`** (DNS/connection-refused) classified non-transient; inner-exception recursion is unreachable for the most common transient storage failure shape. (`PayloadCaptureFailureClassifier.cs:13`)
- **Client abort after response suppresses the audit record silently, even under FailClosed** — `OperationCanceledException` is excluded from both failure paths, so a caller can deliberately disconnect to keep responses out of the audit trail, with no capture-failure log. (`PayloadCaptureMiddleware.cs:55`)
- **Correlation-id sanitization is lossy** — distinct raw ids can collapse to one sanitized id and co-mingle in the same archive stream on unsigned/dev paths. (`CorrelationContext.cs:36`)
- **No retention/purge for `outbox_messages`** — processed/errored rows (full event payloads) accumulate forever; the blob archive got a cleanup story, the table didn't.
- **No ordering guarantee into subscribers** — `maxConcurrentCalls: 16`, no sessions: `status-changed` can process before `created` for the same order. Harmless in the stubs, but no warning comment for whoever implements the email TODO.
- **Security hygiene**: DAST rule-6 suppression is parameter-scoped but not URL-scoped (a future string-typed `page` param silently inherits it); dev tunnel can expose an `UnsignedDevelopment` API (Microsoft-auth-gated, but worth refusing to start when `Mode != Required`); deprecated `X-XSS-Protection: 1` should be `0`; stale SQL Server example connection string in a Postgres stack.

## What held up under adversarial reading

SQL is parameterized everywhere with no dynamic ORDER BY; owner-scoping and 2FA enforcement are pervasive and convention-tested; the U2 correlation-id contract is live and unbypassable (constant-time HMAC, header-hash binding, CR/LF rejection — verified end to end); `.act.env` contains no secrets; supply chain (CPM, locked mode, SHA-pinned actions, source mapping) is exemplary; the outbox claim path (`SKIP LOCKED`, short transactions, locks released before I/O) is correct; DbMigrator matches the EF model column-for-column; Money/Email/state-machine domain logic and the CreateOrder oversell defenses are sound; the two accepted gateway-assertion risks haven't drifted.

## Overall

The pattern across all four reviews is the same — every guarantee holds exactly as far as the test infrastructure can see. The emulator always runs with a healthy consumer (hiding #1, #2), the in-memory store hides #6, #8, and the cleanup divergence, and the convention fleet doesn't inspect the endpoint layer (#9). The fixes are individually small (TTL/dead-letter properties, a host.json retry block, a `ValidateOnStart`, per-row outbox saves); the more durable wins would be a convention test for endpoint `CancellationToken` forwarding and at least one integration test that exercises consumer-downtime and real-host trigger resolution.
